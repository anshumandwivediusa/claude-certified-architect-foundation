# Claude Code Agentic Loop

## Agentic Architecture Context
- **Agentic AI** → Autonomous, Goal Oriented, Dynamic Planning and Reasoning, Systems that Goal → Plan → Reason → Act → Evaluate → Adapt → Repeat → Complete.  
- **Difference from chatbots** → not just one-shot responses; agents iterate until a goal is achieved.  
- **Claude Code** → implements this via orchestration: reasoning engine + execution loop + tools + memory.

## Roles in Agentic AI

### User
- **Who they are** → The human providing instructions, queries, or goals.  
- **Function** → Defines the *objective* (e.g., “Validate customer CUST123 with account ACC456”).  
- **Interaction style** → Natural language, sometimes structured prompts.  
- **Analogy** → The *client* in a service request.


### System
- **Who it is** → The hidden orchestration layer that sets rules, policies, and context.  
- **Function** → Provides constraints, safety, and environment setup.  
- **Examples** →  
  - “You are a helpful assistant.”  
  - **Guardrails like refusal policies, max token limits, or tool access restrictions**.  
- **Analogy** → The ***operating manual***, ***governance framework*** or ***repetative instructions*** that ensures the agent behaves correctly.


### Assistant
- **Who it is** → The AI agent itself (Claude in your code).  
- **Function** →  
  - Interprets user goals.  
  - Decides whether to generate text or invoke tools.  
  - Maintains memory (`messages`).  
  - Iterates until the task is complete.  
- **Analogy** → The *problem solver* or *execution engine*.


### How They Work Together
1. **User** → sets the goal (“Check weather in Austin, then write a haiku”).  
2. **System** → enforces rules (e.g., tool schemas, safety constraints).  
3. **Assistant** → reasons, plans, executes tools, evaluates results, and produces output.  
4. **Loop continues** until the assistant reaches a valid end state or requires human intervention.

### Key Concept
- **User = Goal setter**  
- **System = Rule enforcer**  
- **Assistant = Autonomous executor**  

Together, they form the **agentic triangle** that makes modern AI workflows safe, adaptive, and goal-driven.

## Example : Single Method

```
import anthropic

client = anthropic.Anthropic()

# Define banking tool
tools = [
    {
        "name": "validate_customer",
        "description": "Validate customer account and KYC status",
        "input_schema": {
            "type": "object",
            "properties": {
                "customer_id": {"type": "string"},
                "account_number": {"type": "string"}
            },
            "required": ["customer_id", "account_number"],
        },
    }
]

# Tool execution logic
def execute_tool(name, tool_input):
    if name == "validate_customer":
        # Simulated validation logic
        if tool_input["customer_id"] == "CUST123" and tool_input["account_number"] == "ACC456":
            return "Customer is valid: KYC complete, account active, no fraud flags."
        else:
            return "Customer validation failed: account inactive or KYC pending."
    return "unknown tool"

# Initial user request
messages = [{"role": "user", "content": "Validate customer CUST123 with account ACC456"}]

# Agentic execution loop
while True:
    response = client.messages.create(
        model="claude-sonnet-5",
        max_tokens=1024,
        tools=tools,
        messages=messages,
    )

    messages.append({"role": "assistant", "content": response.content})

    match response.stop_reason:

        case "tool_use":
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    result = execute_tool(block.name, block.input)
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": result,
                    })
            messages.append({"role": "user", "content": tool_results})
            continue

        case "pause_turn":
            continue

        case "end_turn" | "stop_sequence":
            final_text = "".join(b.text for b in response.content if b.type == "text")
            print(final_text)
            break

        case "max_tokens":
            print("Truncated — consider raising max_tokens or summarizing history.")
            break

        case "refusal":
            print("Refused:", response.stop_reason, getattr(response, "stop_details", None))
            messages = messages[:1]
            break

        case "model_context_window_exceeded":
            print("Hit context window — trim or summarize `messages` and retry.")
            break

```

##### Section Note — Tool vs Knowledge in Agentic AI

- **LLM knowledge** → Claude can directly answer general questions (e.g., “KYC = Know Your Customer, a regulatory identity verification process”).  
- **Tool invocation** → If a tool is defined (`whatIsKYC`), Claude may prefer calling it for **authoritative, structured, or policy-grounded answers**.  
- **Decision logic** →  
  - If the query is **generic** → respond directly with text.  
  - If the query matches a **tool schema** → emit `tool_use`.  
  - If both apply → Claude often **calls the tool first**, then enriches with its own knowledge.  
- **stop_reason** →  
  - `"end_turn"` → answered directly from knowledge.  
  - `"tool_use"` → invoked the tool.  
  - Hybrid → `"tool_use"` followed by `"end_turn"` after tool_result.  
- **Best practice** → Use tools for **specific, dynamic, or compliance-critical tasks**, and let LLM knowledge provide **contextual explanation**.

### 🧠 Conceptual Flow
1. **User goal** → Validate customer.  
2. **Assistant reasoning** → Emits `tool_use`.  
3. **Tool execution** → Returns `tool_result`.  
4. **Assistant synthesis** → Emits final `text`.  
5. **Stop reason** → `"end_turn"` signals safe termination.


### 🎯 Key Takeaway
- **Top-level metadata** (`id`, `model`, `stop_reason`, `usage`) → controls orchestration.  
- **Content blocks** (`text`, `tool_use`, `tool_result`) → carry the actual conversation payload.  
- The agent alternates between **tool_use → tool_result → text**, until `stop_reason = "end_turn"`.



## Code Components
- **Anthropic client** → entry point to interact with Claude (`client = anthropic.Anthropic()`).
- **Tools** → external functions the agent can call.  
  - Example: `get_weather` with schema validation.  
  - Tools extend the agent beyond its parameters.
- **Execution function** → `execute_tool` runs the tool logic and returns results.  
- **Messages** → conversation state; appends user + assistant turns for memory.


## Execution Loop
- **Continuous cycle (`while True`)** → agent keeps reasoning until termination.  
- **Response handling (`stop_reason`)** → defines agent’s next step:
  - **tool_use** → agent invokes external tool, returns result, loops again.  
  - **pause_turn** → server-side tool loop reached cap; resend without new input.  
  - **end_turn / stop_sequence** → agent outputs final text and halts.  
  - **max_tokens** → context too long; requires summarization or higher token limit.  
  - **refusal** → policy block; reset context.  
  - **model_context_window_exceeded** → memory overflow; trim or summarize.

## Agentic Concepts in Action
- **Reasoning engine** → Claude decides whether to generate text or call a tool.  
- **Planning** → decomposes “weather + haiku” into subtasks.  
- **Tool invocation** → executes `get_weather`.  
- **Validation** → integrates tool results back into conversation.  
- **Memory** → `messages` list preserves context across iterations.  
- **Feedback loop** → agent adapts strategy until completion.  
- **Termination conditions** → safe exit points ensure reliability.


## Design Principles Reflected
- **Separation of reasoning & execution** → Claude decides, Python executes.  
- **Explicit state** → `messages` maintains structured context.  
- **Fail safely** → handles refusals, truncations, context overflow.  
- **Iterative optimization** → loop continues until success or stop condition.  
- **Controlled autonomy** → agent acts independently but within defined guardrails.

## Tool Use Notes

### What is `tool_use`?
- **Definition:** A `stop_reason` emitted when Claude decides to invoke an external tool defined in the orchestration schema.  
- **Role:** Extends Claude’s capabilities beyond its internal knowledge by executing structured, authoritative functions.  
- **Architect’s Responsibility:** Ensure safe execution, schema compliance, and integration of tool results into the agentic loop.

### Workflow with `tool_use`
1. **User Prompt:** Provides a goal (e.g., “Validate customer CUST123 with ACC456”).  
2. **Claude Response:** Emits `tool_use` with tool name + structured input.  
3. **Execution:** Python (or external system) runs the tool logic.  
4. **Tool Result:** Returned to Claude as `tool_result`.  
5. **Loop:** Claude consumes result, may continue reasoning, or finalize with `end_turn`.


### Architect-Level Complex Examples

#### Example 1: **Parallel Tool Calls**
- **Scenario:** User asks, “Validate customer CUST123 and check weather in Austin.”  
- **Claude Response:** Emits two `tool_use` blocks: `validate_customer` and `get_weather`.  
- **Architect Action:**  
  - Execute both tools in parallel.  
  - Merge results into `messages`.  
  - Ensure Claude integrates both outputs in final synthesis.  
- **Complex Consideration:** Avoid race conditions or partial merges. Architect must design **parallel execution pipelines**.


#### Example 2: **Tool vs Knowledge Conflict**
- **Scenario:** User asks, “What is KYC?” while a `whatIsKYC` tool exists.  
- **Claude Response:** Emits `tool_use` for `whatIsKYC`.  
- **Tool Result:** “KYC = Regulatory identity verification process.”  
- **Claude Knowledge:** Also knows KYC definition.  
- **Architect Action:**  
  - Prioritize tool result as authoritative.  
  - Allow Claude to enrich with contextual explanation.  
- **Complex Consideration:** Architect must enforce **tool-first policy** in compliance-critical workflows.


#### Example 3: **Chained Tool Use**
- **Scenario:** User asks, “Validate customer, then generate compliance summary.”  
- **Claude Response:**  
  - Step 1: `tool_use` → `validate_customer`.  
  - Step 2: Consumes tool_result → emits `end_turn` with compliance summary.  
- **Architect Action:**  
  - Ensure tool_result is appended correctly.  
  - Verify final text integrates tool output.  
- **Complex Consideration:** Architect must design **multi-step chaining** where tool outputs feed into subsequent reasoning.


#### Example 4: **Error Handling in Tool Use**
- **Scenario:** Tool schema mismatch (e.g., missing `account_number`).  
- **Claude Response:** Emits `tool_use` but input invalid.  
- **Architect Action:**  
  - Catch schema error.  
  - Return structured error message.  
  - Allow Claude to re-prompt or request missing input.  
- **Complex Consideration:** Architect must implement **schema validation + recovery loop**.


## Key Architect Insights
- **Tool_use = authoritative execution path.**  
- **Parallel execution** requires careful orchestration.  
- **Tool-first policy** ensures compliance and reliability.  
- **Chained tool calls** enable multi-step workflows.  
- **Schema validation** prevents runtime failures.  
- **Integration with memory (`messages`)** ensures continuity across iterations.
