# Claude Code Agentic Loop

## Agentic Architecture Context
- **Agentic AI** ‚Üí Autonomous, Goal Oriented, Dynamic Planning and Reasoning, Systems that Goal ‚Üí Plan ‚Üí Reason ‚Üí Act ‚Üí Evaluate ‚Üí Adapt ‚Üí Repeat ‚Üí Complete.  
- **Difference from chatbots** ‚Üí not just one-shot responses; agents iterate until a goal is achieved.  
- **Claude Code** ‚Üí implements this via orchestration: reasoning engine + execution loop + tools + memory.

## Roles in Agentic AI

### User
- **Who they are** ‚Üí The human providing instructions, queries, or goals.  
- **Function** ‚Üí Defines the *objective* (e.g., ‚ÄúValidate customer CUST123 with account ACC456‚Äù).  
- **Interaction style** ‚Üí Natural language, sometimes structured prompts.  
- **Analogy** ‚Üí The *client* in a service request.


### System
- **Who it is** ‚Üí The hidden orchestration layer that sets rules, policies, and context.  
- **Function** ‚Üí Provides constraints, safety, and environment setup.  
- **Examples** ‚Üí  
  - ‚ÄúYou are a helpful assistant.‚Äù  
  - **Guardrails like refusal policies, max token limits, or tool access restrictions**.  
- **Analogy** ‚Üí The ***operating manual***, ***governance framework*** or ***repetative instructions*** that ensures the agent behaves correctly.


### Assistant
- **Who it is** ‚Üí The AI agent itself (Claude in your code).  
- **Function** ‚Üí  
  - Interprets user goals.  
  - Decides whether to generate text or invoke tools.  
  - Maintains memory (`messages`).  
  - Iterates until the task is complete.  
- **Analogy** ‚Üí The *problem solver* or *execution engine*.


### How They Work Together
1. **User** ‚Üí sets the goal (‚ÄúCheck weather in Austin, then write a haiku‚Äù).  
2. **System** ‚Üí enforces rules (e.g., tool schemas, safety constraints).  
3. **Assistant** ‚Üí reasons, plans, executes tools, evaluates results, and produces output.  
4. **Loop continues** until the assistant reaches a valid end state or requires human intervention.

### Key Concept
- **User = Goal setter**  
- **System = Rule enforcer**  
- **Assistant = Autonomous executor**  

Together, they form the **agentic triangle** that makes modern AI workflows safe, adaptive, and goal-driven.

## Example : Single Method

```
import anthropic

client = anthropic.Anthropic()

# Define banking tool
tools = [
    {
        "name": "validate_customer",
        "description": "Validate customer account and KYC status",
        "input_schema": {
            "type": "object",
            "properties": {
                "customer_id": {"type": "string"},
                "account_number": {"type": "string"}
            },
            "required": ["customer_id", "account_number"],
        },
    }
]

# Tool execution logic
def execute_tool(name, tool_input):
    if name == "validate_customer":
        # Simulated validation logic
        if tool_input["customer_id"] == "CUST123" and tool_input["account_number"] == "ACC456":
            return "Customer is valid: KYC complete, account active, no fraud flags."
        else:
            return "Customer validation failed: account inactive or KYC pending."
    return "unknown tool"

# Initial user request
messages = [{"role": "user", "content": "Validate customer CUST123 with account ACC456"}]

# Agentic execution loop
while True:
    response = client.messages.create(
        model="claude-sonnet-5",
        max_tokens=1024,
        tools=tools,
        messages=messages,
    )

    messages.append({"role": "assistant", "content": response.content})

    match response.stop_reason:

        case "tool_use":
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    result = execute_tool(block.name, block.input)
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": result,
                    })
            messages.append({"role": "user", "content": tool_results})
            continue

        case "pause_turn":
            continue

        case "end_turn" | "stop_sequence":
            final_text = "".join(b.text for b in response.content if b.type == "text")
            print(final_text)
            break

        case "max_tokens":
            print("Truncated ‚Äî consider raising max_tokens or summarizing history.")
            break

        case "refusal":
            print("Refused:", response.stop_reason, getattr(response, "stop_details", None))
            messages = messages[:1]
            break

        case "model_context_window_exceeded":
            print("Hit context window ‚Äî trim or summarize `messages` and retry.")
            break

```

## Execution Loop

- **Response handling (`stop_reason`)** ‚Üí defines agent‚Äôs next step:

  - **max_tokens** ‚Üí context too long; requires summarization or higher token limit.  
  - **refusal** ‚Üí policy block; reset context.  
  - **model_context_window_exceeded** ‚Üí memory overflow; trim or summarize.

## Agentic Concepts in Action
- **Reasoning engine** ‚Üí Claude decides whether to generate text or call a tool.  
- **Planning** ‚Üí decomposes ‚Äúweather + haiku‚Äù into subtasks.  
- **Tool invocation** ‚Üí executes `get_weather`.  
- **Validation** ‚Üí integrates tool results back into conversation.

- **Feedback loop** ‚Üí agent adapts strategy until completion.

## Design Principles Reflected
- **Separation of reasoning & execution** ‚Üí Claude decides, Python executes.  
- **Explicit state** ‚Üí `messages` maintains structured context.  
- **Fail safely** ‚Üí handles refusals, truncations, context overflow.  
- **Iterative optimization** ‚Üí loop continues until success or stop condition.  
- **Controlled autonomy** ‚Üí agent acts independently but within defined guardrails.

## Tool Use Notes

### What is `tool_use`?
- **Definition:** A `stop_reason` emitted when Claude decides to invoke an external tool defined in the orchestration schema.  
- **Role:** Extends Claude‚Äôs capabilities beyond its internal knowledge by executing structured, authoritative functions.  
- **Architect‚Äôs Responsibility:** Ensure safe execution, schema compliance, and integration of tool results into the agentic loop.

### Workflow with `tool_use`
1. **User Prompt:** Provides a goal (e.g., ‚ÄúValidate customer CUST123 with ACC456‚Äù).  
2. **Claude Response:** Emits `tool_use` with tool name + structured input.  
3. **Execution:** Python (or external system) runs the tool logic.  
4. **Tool Result:** Returned to Claude as `tool_result`.  
5. **Loop:** Claude consumes result, may continue reasoning, or finalize with `end_turn`.

### Architect-Level Complex Examples

#### Example 1: **Parallel Tool Calls**
- **Scenario:** User asks, ‚ÄúValidate customer CUST123 and check weather in Austin.‚Äù  
- **Claude Response:** Emits two `tool_use` blocks: `validate_customer` and `get_weather`.  
- **Architect Action:**  
  - Execute both tools in parallel.  
  - Merge results into `messages`.  
  - Ensure Claude integrates both outputs in final synthesis.  
- **Complex Consideration:** Avoid race conditions or partial merges. Architect must design **parallel execution pipelines**.


```
import anthropic
import concurrent.futures

client = anthropic.Anthropic()

# Define tools
tools = [
    {
        "name": "validate_customer",
        "description": "Validate customer account and KYC status",
        "input_schema": {
            "type": "object",
            "properties": {
                "customer_id": {"type": "string"},
                "account_number": {"type": "string"}
            },
            "required": ["customer_id", "account_number"],
        },
    },
    {
        "name": "get_weather",
        "description": "Fetch current weather for a city",
        "input_schema": {
            "type": "object",
            "properties": {
                "city": {"type": "string"}
            },
            "required": ["city"],
        },
    }
]

# Tool execution logic
def execute_tool(name, tool_input):
    if name == "validate_customer":
        if tool_input["customer_id"] == "CUST123" and tool_input["account_number"] == "ACC456":
            return "Customer valid: KYC complete, account active."
        else:
            return "Customer validation failed."
    elif name == "get_weather":
        if tool_input["city"].lower() == "austin":
            return "Austin weather: 32¬∞C, sunny."
        else:
            return f"Weather data not available for {tool_input['city']}."
    return "Unknown tool"

# Initial user request
messages = [{"role": "user", "content": "Validate customer CUST123 with ACC456 and check weather in Austin"}]

# Agentic execution loop
while True:
    response = client.messages.create(
        model="claude-sonnet-5",
        max_tokens=1024,
        tools=tools,
        messages=messages,
    )

    messages.append({"role": "assistant", "content": response.content})

    match response.stop_reason:
        case "tool_use":
            tool_results = []

            # Run tool calls in parallel
            with concurrent.futures.ThreadPoolExecutor() as executor:
                futures = []
                for block in response.content:
                    if block.type == "tool_use":
                        futures.append(
                            executor.submit(execute_tool, block.name, block.input)
                        )
                for i, future in enumerate(futures):
                    result = future.result()
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": response.content[i].id,
                        "content": result,
                    })

            # Merge results back into messages
            messages.append({"role": "user", "content": tool_results})
            continue

        case "end_turn":
            final_text = "".join(b.text for b in response.content if b.type == "text")
            print("Final Output:", final_text)
            break

        case _:
            print("Stop reason:", response.stop_reason)
            break

```
- **Complex Consideration:** Architect-Level Strategies
  - Thread-safe Execution: Use ThreadPoolExecutor or async coroutines to run tools in parallel.

  - Deterministic Merging: Collect results in a structured list keyed by tool_use_id.

  - Schema Validation: Ensure each tool input matches its schema before execution.

  - Error Isolation: If one tool fails, return structured error without blocking others.

  - Message Integration: Append merged results as a single messages entry to avoid partial state.

## `end_turn` Stop Reason

### What is `end_turn`?
- **Definition:** A `stop_reason` emitted when Claude has **completed its task** and produced the final output.  
- **Role:** Signals safe termination of the agentic loop.  
- **Architect‚Äôs Responsibility:**  
  - Detect `end_turn`.  
  - Present final text output to the user.  
  - Halt the loop (no further tool calls or retries).  


### Workflow with `end_turn`
1. **User Prompt:** ‚ÄúValidate customer CUST123 with ACC456.‚Äù  
2. **Claude Response:** Emits `tool_use` ‚Üí calls `validate_customer`.  
3. **Tool Result:** ‚ÄúCustomer valid: KYC complete.‚Äù  
4. **Claude Next Step:** Emits `end_turn` with final text:  
   ```
   Validation complete. The customer is active, KYC-compliant, and cleared of fraud flags.
   ```
5. **Architect Action:** Stop loop, present final text.


### Example Response Structure

```json
{
  "id": "msg_002",
  "model": "claude-sonnet-5",
  "stop_reason": "end_turn",
  "role": "assistant",
  "content": [
    {
      "type": "text",
      "text": "Validation complete. The customer is active, KYC-compliant, and cleared of fraud flags."
    }
  ],
  "usage": {
    "input_tokens": 60,
    "output_tokens": 30
  }
}
```


### Architect-Level Section Notes

- **Finalization:**  
  `end_turn` means Claude has synthesized tool results and produced the final text.  
  Architect must stop the loop and deliver output.

- **Hybrid Flows:**  
  Often follows `tool_use` ‚Üí `tool_result` ‚Üí `end_turn`.  
  Architect ensures tool outputs are integrated into final synthesis.

- **Safe Termination:**  
  No further retries or tool calls. Architect halts execution cleanly.

- **Logging:**  
  Architect may log `end_turn` events to confirm workflow completion.

## `pause_turn` Stop Reason

### What is `pause_turn`?
- **Definition:** A `stop_reason` emitted when the **server-side tool loop cap** is reached.  
- **Role:** Signals that Claude has paused execution because it hit the maximum allowed tool iterations in one loop.  
- **Architect‚Äôs Responsibility:**  
  - Detect `pause_turn`.  
  - Resend the request **without new user input** to continue execution.  
  - Avoid duplication of tool calls or partial state merges.


### Workflow with `pause_turn`
1. **User Prompt:** ‚ÄúValidate customer, check weather, then generate compliance summary.‚Äù  
2. **Claude Response:** Emits multiple `tool_use` calls in sequence.  
3. **Server Loop:** After hitting cap (e.g., 5 tool calls), Claude emits `pause_turn`.  
4. **Architect Action:**  
   - Resend request with current `messages`.  
   - Do not add new user input.  
   - Allow Claude to continue reasoning from paused state.


### Example Response

```json
{
  "id": "msg_004",
  "model": "claude-sonnet-5",
  "stop_reason": "pause_turn",
  "role": "assistant",
  "content": [
    {
      "type": "text",
      "text": "Pausing execution after multiple tool calls. Continuing..."
    }
  ]
}
```


### Architect-Level Section Notes

- **Loop Cap:**  
  `pause_turn` occurs when Claude reaches the server-side iteration cap for tool calls.  

- **Resend Without Input:**  
  Architect must resend the request with the same `messages` to continue. Adding new input risks duplication.  

- **Avoid Duplication:**  
  If architect mistakenly adds new user input, Claude may re-trigger the same tool calls.  

- **State Continuity:**  
  Messages must preserve prior tool_use and tool_result so Claude can resume correctly.  


### Complex Examples

#### Example 1: **Multi-Tool Workflow**

- **Architect Action:** Resend with same `messages`. Claude resumes with `end_turn`.  


#### Example 2: **Error Isolation**
